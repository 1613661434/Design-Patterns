# 命令模式 (Command Pattern)

## 一、模式概述

**命令模式**是一种行为型设计模式，它将请求封装为对象，使客户端可以用不同的请求参数化对象，并且支持请求的排队、记录、撤销等操作。该模式通过解耦请求发送者和接收者，使系统更灵活地处理各种操作。

## 二、结构组成

| 角色 | 说明 |
|------|------|
| **接收者 (Receiver)** | 执行请求的具体对象（`CookerZeff`） |
| **抽象命令 (Command)** | 定义请求的接口，包含执行方法（`AbstractCommand`） |
| **具体命令 (Concrete Command)** | 实现抽象命令，绑定接收者的具体操作（`DSXCommand`、`GBJDCommand`等） |
| **调用者 (Invoker)** | 发起请求，持有命令对象并执行其方法（`WaiterLuffy`） |

## 三、海贼王场景示例

以餐厅点餐系统为例：顾客通过服务员路飞（调用者）向厨师哲普（接收者）下单，每个菜品对应一个命令对象，服务员负责管理订单（添加、取消、执行）。

- **CookerZeff (厨师哲普)**: 实际做菜的接收者
- **AbstractCommand**: 抽象命令，封装一个菜品订单
- **DSXCommand / GBJDCommand / YXRSCommand / HSPGCommand**: 地三鲜、宫保鸡丁、鱼香肉丝、红烧排骨命令
- **WaiterLuffy (服务员路飞)**: 管理订单的下单、撤销、通知后厨、结账

## 四、核心代码结构

```
include/
  ├── Command.hpp        # 接收者 + 抽象命令 + 具体命令
  └── Waiter.hpp         # 调用者（服务员）
src/
  └── main.cpp           # 测试入口
```

## 五、优缺点

### 优点
- **解耦请求发送者和接收者**: 发送者不需要知道接收者如何执行
- **可扩展性好**: 新增命令不需要修改现有代码
- **支持撤销/重做**: 命令对象可以记录状态，支持回滚
- **支持宏命令**: 可以将多个命令组合成一个复合命令

### 缺点
- **类的数量增加**: 每个具体命令都需要一个类
- **系统复杂度增加**: 引入了额外的抽象层

## 六、适用场景

- 需要参数化对象以执行操作（如回调函数）
- 需要支持操作的撤销/重做
- 需要将操作排队、记录日志或支持事务
- GUI 按钮、菜单项等需要与具体操作解耦的场景
