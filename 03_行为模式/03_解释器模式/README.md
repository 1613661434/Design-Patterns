# 解释器模式 (Interpreter Pattern)

## 一、模式概述

**解释器模式**是一种行为型设计模式，它定义了一种语言的语法表示，并提供解释器来处理该语法。该模式将语法规则表示为类层次结构，每个语法规则对应一个表达式类，使语言的解释过程更加灵活和可扩展。

## 二、结构组成

| 角色 | 说明 |
|------|------|
| **抽象表达式 (Abstract Expression)** | 定义解释操作的接口（`Expression`） |
| **终结符表达式 (Terminal Expression)** | 实现与文法中的终结符相关的解释操作（`NumberExpression`） |
| **非终结符表达式 (Non-terminal Expression)** | 实现文法中非终结符的解释操作（`AddExpression`、`SubtractExpression`） |
| **上下文 (Context)** | 包含解释器需要的全局信息（示例中隐式使用） |

## 三、场景示例

以简单的算术表达式解释器为例：使用解释器模式解析并计算形如 `(3 + (5 - 2)) + 7` 的表达式。每个操作符（如 `+`、`-`）和操作数（如 `3`、`5`）都对应一个表达式类，通过组合这些类构建表达式树并求值。

- **NumberExpression**: 终结符，代表具体数字
- **AddExpression**: 非终结符，代表加法运算，包含左右两个子表达式
- **SubtractExpression**: 非终结符，代表减法运算

## 四、核心代码结构

```
include/
  └── Expression.hpp    # 抽象表达式 + 具体表达式类
src/
  └── main.cpp          # 构建表达式树 + 求值
```

## 五、优缺点

### 优点
- **易于扩展语法**: 增加新的表达式类即可扩展语言
- **易于实现**: 每个语法规则对应一个类，结构清晰
- **表达式树表示**: 自然地用组合模式表示复杂表达式

### 缺点
- **类的数量膨胀**: 每个语法规则都需要一个类，复杂语法会导致类爆炸
- **执行效率低**: 递归调用和大量对象创建影响性能
- **适用场景有限**: 只适合简单语法的场景

## 六、适用场景

- 有一种简单的语言或表达式需要解释执行
- 语法规则较少，相对稳定，且执行效率要求不高
- SQL 解析、正则表达式匹配、数学公式求值等
- 将频繁变化的业务规则用解释器表示
