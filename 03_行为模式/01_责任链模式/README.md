# 责任链模式 (Chain of Responsibility Pattern)

## 一、模式概述

**责任链模式**是一种行为型设计模式，它将处理请求的对象连成一条链，当请求发送时，链上的对象依次处理请求，直到有一个对象处理它为止。该模式避免了请求发送者和接收者之间的耦合，使多个对象都有机会处理请求，提高了系统的灵活性。

## 二、结构组成

| 角色 | 说明 |
|------|------|
| **抽象处理者 (Handler)** | 定义处理请求的接口，包含设置下一个处理者的方法（`AbstractManager`） |
| **具体处理者 (Concrete Handler)** | 实现抽象处理者接口，处理请求或传递给下一个处理者（`Manager`、`CEO`、`Boss`） |
| **客户端 (Client)** | 发送请求到责任链（`DaCongMing`） |

## 三、海贼王场景示例

以公司员工的请求处理为例：员工（大聪明）向公司提出请假、涨薪或辞职的请求，请求依次经过**部门经理**、**CEO**、**老板**的处理，每个管理者根据权限决定处理请求或传递给下一级。

- **Manager (部门经理)**: 只能批请假，涨薪和辞职需上报
- **CEO**: 能批请假和涨薪（给购物券），辞职需上报老板
- **Boss (老板)**: 所有请求都能最终处理（但不一定同意）

## 四、核心代码结构

```
include/
  └── Manager.hpp        # 抽象处理者 + 具体处理者类
src/
  └── main.cpp           # 具体实现 + 测试入口
```

## 五、优缺点

### 优点
- **降低耦合度**: 请求发送者和接收者解耦，发送者无需知道谁处理了请求
- **灵活性高**: 可动态增加、删除或重新排列处理者
- **单一职责**: 每个处理者只处理自己负责的请求类型

### 缺点
- **请求可能被遗漏**: 如果没有正确配置责任链，请求可能得不到处理
- **性能开销**: 请求可能需要遍历整条链才被处理
- **调试困难**: 链式调用的调试和排错比较困难

## 六、适用场景

- 多个对象可以处理同一请求，但具体由哪个对象处理在运行时决定
- 需要向多个对象中的一个提交请求，而不明确指定接收者
- 需要动态指定处理请求的对象集合（如审批流程、拦截器链等）
