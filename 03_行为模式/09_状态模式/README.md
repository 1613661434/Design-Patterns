# 状态模式 (State Pattern)

## 一、模式概述

**状态模式**是一种行为型设计模式，它允许对象在内部状态改变时改变其行为，使对象看起来像是修改了自身的类。该模式将状态封装为独立的类，通过状态之间的转换来改变对象的行为，避免使用大量条件判断语句。

## 二、结构组成

| 角色 | 说明 |
|------|------|
| **抽象状态 (State)** | 定义状态相关的接口或抽象类（`AbstractState`） |
| **具体状态 (Concrete State)** | 实现抽象状态接口，定义特定状态下的行为（`ForenoonState`、`NoonState`等） |
| **上下文 (Context)** | 持有当前状态的引用，根据状态调用相应行为（`Sanji`） |

## 三、海贼王场景示例

以《海贼王》中桑吉的一天为例：桑吉根据时间（上午、中午、下午、晚上）切换不同的状态，每个状态对应不同的行为。

- **ForenoonState (上午)**: <8点准备早餐，8-11点钓鱼
- **NoonState (中午)**: <13点做午饭，之后自动切换到下午
- **AfternoonState (下午)**: <15点做甜点，15-18点钓鱼，之后切换到晚上
- **EveningState (晚上)**: <19点做晚饭，之后睡觉
- **Sanji (桑吉)**: 持有当前时间（clock）和当前状态

## 四、核心代码结构

```
include/
  ├── State.hpp          # AbstractState + 4个具体状态声明（前向声明 Sanji）
  └── Sanji.hpp          # Sanji 上下文类
src/
  └── main.cpp           # 所有 working() 实现 + 测试入口
```

## 五、优缺点

### 优点
- **消除条件分支**: 用多态替代大量 if-else 或 switch
- **状态转换显式化**: 状态转换逻辑集中在具体状态类中
- **开闭原则**: 增加新状态不影响现有代码
- **状态可复用**: 状态对象可在不同上下文中复用

### 缺点
- **类爆炸**: 状态多了会导致类数量急剧增加
- **状态转换复杂**: 多个状态之间的转换关系可能难以理解
- **上下文与状态耦合**: 状态类需要知道上下文的接口

## 六、适用场景

- 对象的行为随状态变化而变化
- 代码中有大量与状态相关的条件判断
- 状态数量较多且可能频繁变化
- 工作流引擎、游戏角色状态机、订单流转等
