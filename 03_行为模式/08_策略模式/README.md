# 策略模式 (Strategy Pattern)

## 一、模式概述

**策略模式**是一种行为型设计模式，它定义了一系列算法，并将每个算法封装起来，使它们可以相互替换。该模式让算法的变化独立于使用算法的客户端，适用于需要在运行时动态切换算法的场景。

## 二、结构组成

| 角色 | 说明 |
|------|------|
| **抽象策略 (Strategy)** | 定义所有支持算法的公共接口（`AbstractStrategy`） |
| **具体策略 (Concrete Strategy)** | 实现抽象策略接口的具体算法（`YiDang`、`ErDang`等） |
| **上下文 (Context)** | 持有一个策略对象的引用，负责选择和使用策略（`Luffy`） |

## 三、海贼王场景示例

以《海贼王》中路飞的战斗模式为例：路飞根据敌人的强度动态切换战斗策略（一档到五档），每种策略对应不同的攻击方式，体现了算法的动态切换。

- **YiDang (一档)**: 基础橡胶招式（Easy）
- **ErDang (二挡)**: 血液加速，Jet系列招式（Normal）
- **SanDang (三挡)**: 巨人化攻击（Hard）
- **SiDang (四挡)**: 武装色强化，狮子/犀牛炮弹（Experts）
- **WuDang (五档)**: 尼卡形态，任意改变物体形态（Professional）
- **Luffy (路飞)**: 根据 Level 枚举动态选择档位

## 四、核心代码结构

```
include/
  ├── Strategy.hpp       # 抽象策略 + 5种具体策略
  └── Luffy.hpp          # Luffy 上下文类 + Level 枚举
src/
  └── main.cpp           # 测试入口
```

## 五、优缺点

### 优点
- **算法可自由切换**: 运行时动态选择策略
- **避免多重条件判断**: 用多态替代 switch/if-else
- **扩展性好**: 新增策略无需修改上下文
- **开闭原则**: 对扩展开放，对修改封闭

### 缺点
- **类数量增加**: 每个策略都要一个类
- **客户端必须了解策略差异**: 需要知道有哪些策略可以选择
- **策略类之间完全独立**: 无法共享数据

## 六、适用场景

- 需要在运行时动态切换算法或行为
- 类有多种行为，通过大量 if-else 或 switch 选择
- 不希望客户端知道复杂的算法数据结构
- 支付方式选择、排序算法切换、压缩算法选择等
