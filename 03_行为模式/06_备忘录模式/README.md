# 备忘录模式 (Memento Pattern)

## 一、模式概述

**备忘录模式**是一种行为型设计模式，它允许在不暴露对象内部状态的前提下，捕获并保存对象的状态，以便后续将对象恢复到保存的状态。该模式通过分离状态的保存和管理，实现了对象状态的快照机制，常用于需要回滚操作或保存历史状态的场景。

## 二、结构组成

| 角色 | 说明 |
|------|------|
| **原发器 (Originator)** | 创建并恢复状态的对象（`JiaoPenJi`） |
| **备忘录 (Memento)** | 保存原发器的状态，确保状态不被外部修改（`History`） |
| **管理者 (Caretaker)** | 存储和管理备忘录，不直接操作备忘录的内容（`Recorder`） |

## 三、海贼王/现实场景示例

以"脚盆鸡（日本）承认历史罪行"为例：原发器对象（脚盆鸡）有多个不承认历史罪行的状态，通过备忘录记录这些状态，当被"兔子（中国）"教训后，被迫依次承认这些罪行，实现状态的恢复和展示。

- **JiaoPenJi (脚盆鸡/原发器)**: 持有历史认知状态（不承认各种罪行）
- **History (备忘录)**: 不可变的备忘录对象，封装状态快照
- **Recorder (记录者/管理者)**: 保存多个备忘录，按索引检索

## 四、核心代码结构

```
include/
  ├── Memento.hpp        # History (备忘录) + JiaoPenJi (原发器)
  └── Recorder.hpp       # Recorder (管理者)
src/
  └── main.cpp           # 测试入口
```

## 五、优缺点

### 优点
- **封装性好**: 备忘录只能由原发器访问和恢复，外部不可修改
- **简化原发器**: 状态保存和恢复的工作由备忘录和管理者分担
- **支持撤销/恢复**: 轻松实现撤销操作

### 缺点
- **内存开销大**: 频繁创建备忘录可能消耗大量内存
- **维护成本**: 管理者需要管理备忘录的生命周期
- **动态语言限制**: 部分语言难以保证只有原发器能访问备忘录

## 六、适用场景

- 需要保存对象的历史状态并支持回滚（如 Ctrl+Z 撤销）
- 需要保护状态数据的封装性（外部不能直接访问状态）
- 游戏存档、文本编辑器撤销、数据库事务回滚等
