# 组合模式 (Composite Pattern)

## 一、模式意图 (Intent)

将对象组合成**树形结构**来表示"部分-整体"的层次关系，使客户端对单个对象（叶子节点）和组合对象（组合节点）的操作具有**一致性**。

## 二、模式结构 (Structure)

```
                    ┌──────────────────┐
                    │  AbstractTeam    │  <-- Component (抽象节点)
                    │  + add()         │
                    │  + remove()      │
                    │  + fight()       │
                    │  + display()     │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
     ┌────────┴────────┐    ┌┴─────────────────┐
     │   LeafTeam      │    │   ManagerTeam    │
     │   (Leaf)        │    │   (Composite)    │
     │   无子节点       │    │   - m_children   │
     │   + fight()     │    │   + add()        │
     │   + display()   │    │   + remove()     │
     └─────────────────┘    └────────┬─────────┘
                                     │ 0..*
                            ┌────────┴─────────┐
                            │   AbstractTeam*  │
                            └──────────────────┘
```

| 角色 | 类名 | 说明 |
|------|------|------|
| Component（抽象节点） | AbstractTeam | 定义叶子与组合的公共接口 |
| Leaf（叶子节点） | LeafTeam | 最小作战单元，无子节点 (番队) |
| Composite（组合节点） | ManagerTeam | 包含子节点，管理下级团队 |
| Client（客户端） | main.cpp | 通过Component接口操作对象 |

## 三、示例场景 (海贼王主题)

**草帽大船团战斗编队**: 以草帽海贼团为根节点，下辖7个分团（组合节点），其中最后一个分团"约塔玛利亚大船团"下包含9个番队（叶子节点）。通过组合模式统一处理单个番队和整个团队的战斗逻辑。

```
草帽海贼团 (ManagerTeam - 根节点)
├── 俊美海贼团 (ManagerTeam)
├── 巴托俱乐部 (ManagerTeam)
├── 八宝水军 (ManagerTeam)
├── 艾迪欧海贼团 (ManagerTeam)
├── 咚塔塔海贼团 (ManagerTeam)
├── 巨兵海贼团 (ManagerTeam)
└── 约塔玛利亚大船团 (ManagerTeam)
    ├── 第1番队 (LeafTeam)
    ├── 第2番队 (LeafTeam)
    ├── ...
    └── 第9番队 (LeafTeam)
```

## 四、优缺点

### 优点
1. **一致性**: 客户端可一致地处理单个对象和组合对象
2. **开闭原则**: 新增节点类型无需修改现有客户端代码
3. **简化客户端**: 客户端无需关心处理的是叶子还是组合
4. **树形结构**: 自然表达层次化数据

### 缺点
1. 设计过于一般化，难以限制组合中的组件类型
2. 使设计变得更加抽象，增加理解难度
3. 叶子节点也会有add/remove等无意义的方法（安全性与透明性的权衡）

## 五、适用场景

- 需要表示对象的"部分-整体"层次结构
- 希望客户端忽略单个对象和组合对象的差异
- 需要统一处理树形结构中的所有对象

## 六、透明性 vs 安全性

- **透明组合**: 所有方法定义在Component中（如本例），客户端无需区分节点类型
- **安全组合**: 管理方法只定义在Composite中，更安全但客户端需区分类型
