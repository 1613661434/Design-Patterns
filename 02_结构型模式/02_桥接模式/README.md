# 桥接模式 (Bridge Pattern)

## 一、模式意图 (Intent)

将抽象部分与实现部分分离，使它们都可以独立地变化。该模式通过**组合**而非**继承**的方式连接抽象和实现，避免多层继承导致的类爆炸问题。

## 二、模式结构 (Structure)

```
┌─────────────────┐          ┌─────────────────┐
│  AbstractShip   │          │  AbstractTeam   │
│  (Abstraction)  │─────────>│  (Implementor)  │
│  - m_team*      │  组合     │  + addMember()  │
│  + showTeam()   │          │  + executeTask() │
└────────┬────────┘          └────────┬────────┘
         │                            │
  ┌──────┼──────┐              ┌──────┼──────┐
  │      │      │              │      │      │
Merry  HaiJunShip         CaoMaoTeam  SmokerTeam
(R.A.) (R.A.)            (C.I.)      (C.I.)
```

| 角色 | 类名 | 说明 |
|------|------|------|
| Abstraction（抽象类） | AbstractShip | 定义船的抽象接口，持有团队引用 |
| RefinedAbstraction（扩展抽象） | Merry, HaiJunShip | 具体船型，扩展船的抽象功能 |
| Implementor（实现接口） | AbstractTeam | 定义团队的实现接口 |
| ConcreteImplementor（具体实现） | CaoMaoTeam, SmokerTeam | 具体团队，实现团队的逻辑 |
| Data Class | Person | 人员信息数据类 |

## 三、示例场景 (海贼王主题)

**船与团队的桥接**: 船（梅利号、海军舰）需要搭载不同的团队（草帽海贼团、斯摩格部队）。通过桥接模式将"船"的抽象与"团队"的实现分离，使两者可以独立扩展：

- 新增船型时，无需修改团队类
- 新增团队时，无需修改船类
- 船和团队可以任意组合

## 四、优缺点

### 优点
1. **分离抽象与实现**: 两者可以独立变化和扩展
2. **开闭原则**: 新增抽象或实现无需修改对方
3. **组合优于继承**: 避免多层继承导致的类爆炸
4. **运行时绑定**: 可以在运行时动态选择实现

### 缺点
1. 增加了系统的理解与设计难度
2. 需要正确识别出系统中两个独立变化的维度

## 五、适用场景

- 需要避免抽象与实现之间的紧耦合
- 抽象和实现都需要通过子类进行扩展
- 需要在运行时切换实现
- 一个类存在两个（或多个）独立变化的维度
- 想要在多个对象间共享实现
