# 外观模式 (Facade Pattern)

## 一、模式意图 (Intent)

为复杂子系统提供一个**简单的统一接口**，隐藏系统内部的复杂实现，使客户端可以通过统一的接口访问系统。该模式通过封装子系统的交互逻辑，降低客户端与子系统的耦合度，提高代码的可维护性。

## 二、模式结构 (Structure)

```
┌──────────────────────────────────────────────────────────┐
│                      LionCannon (Facade)                 │
│  + aimAndLock()     // 封装: 可乐注入 + 能量转换 + 目标锁定 │
│  + fire()           // 封装: 开炮 + 风来炮稳定            │
└──────┬───────┬───────┬────────┬─────────┐
       │       │       │        │         │
       v       v       v        v         v
  CokeSystem EnergySystem AimLockSystem Cannon WindCannon
   (Subsystem)(Subsystem) (Subsystem) (Subsystem)(Subsystem)
```

| 角色 | 类名 | 说明 |
|------|------|------|
| Facade（外观类） | LionCannon | 封装子系统，提供简单接口 |
| Subsystem（子系统） | CokeSystem, EnergySystem, AimLockSystem, Cannon, WindCannon | 实现具体功能，相互协作 |

## 三、示例场景 (海贼王主题)

**桑尼号的狮吼炮发射系统**: 发射狮吼炮需要5个子系统按特定顺序协同工作：

1. `CokeSystem`: 注入可乐作为原料
2. `EnergySystem`: 将可乐转换为能量
3. `AimLockSystem`: 瞄准并锁定目标
4. `Cannon`: 发射狮吼炮
5. `WindCannon`: 发射风来炮抵消后坐力

外观类 `LionCannon` 将这些步骤封装为：
- `aimAndLock()`: 执行步骤1-3（瞄准阶段）
- `fire()`: 执行步骤4-5（开炮阶段）

## 四、优缺点

### 优点
1. **简化接口**: 客户端只需与外观交互，无需了解子系统细节
2. **降低耦合**: 子系统变化不影响客户端
3. **更好的分层**: 在多层系统中，外观可以定义各层的入口点
4. **迪米特法则**: 减少客户端需要交互的对象数量

### 缺点
1. 外观类可能成为"上帝对象"，与所有类耦合
2. 不符合开闭原则：增加子系统可能需要修改外观类
3. 外观不能阻止高级用户直接访问子系统

## 五、适用场景

- 需要为复杂子系统提供简单的入口
- 需要将子系统分层，外观作为每一层的入口
- 需要降低客户端与子系统之间的耦合度
- 子系统包含多个复杂的协作步骤，需要按特定顺序执行

## 六、外观 vs 其他模式

| 模式 | 核心行为 |
|------|----------|
| 外观模式 | 封装复杂子系统，提供简化接口 |
| 适配器模式 | 转换不兼容接口 |
| 中介者模式 | 协调多个对象之间的交互 |
