# 代理模式 (Proxy Pattern)

## 一、模式意图 (Intent)

通过创建一个**代理对象**来控制对原始对象的访问。代理对象与原始对象实现相同的接口，客户端通过代理对象间接访问原始对象，从而在不修改原始对象的情况下，添加额外的控制逻辑（如权限验证、缓存、日志记录等）。

## 二、模式结构 (Structure)

```
┌──────────────────────┐
│   Communication      │  <-- Subject (抽象接口)
│   + communicate() = 0│
└──────────┬───────────┘
           │
  ┌────────┴──────────────────┐
  │                           │
┌─┴──────────────┐    ┌──────┴──────────┐
│    Speaker     │    │   DenDenMushi   │
│ (Real Subject) │    │    (Proxy)      │
│  原始通话逻辑    │    │  - m_speaker*   │
│  + communicate()│    │  - m_isStart    │
└────────────────┘    │  + communicate()│  ─── 先模仿, 再委托
                      └────────────────┘
```

| 角色 | 类名 | 说明 |
|------|------|------|
| Subject（抽象接口） | Communication | 定义代理和原始对象的公共接口 |
| RealSubject（原始对象） | Speaker | 实现实际业务逻辑（通话） |
| Proxy（代理对象） | DenDenMushi | 持有原始对象引用，添加控制逻辑（模仿） |

## 三、示例场景 (海贼王主题)

**电话虫代理通话**: 电话虫作为代理，模仿通话者的语言和表情。在原始通话（Speaker）的基础上添加"实时模仿"的功能：

1. `Speaker::communicate()`: 说话+表情变化（原始业务逻辑）
2. `DenDenMushi::communicate()`: 先模仿语言和表情，再委托给Speaker

客户端使用相同的 `Communication` 接口进行交互，无需关心使用的是直接通话还是电话虫。

## 四、代理模式的类型

| 类型 | 用途 | 说明 |
|------|------|------|
| **远程代理** | 访问远程对象 | 为不同地址空间的对象提供本地代表 |
| **虚拟代理** | 延迟加载 | 按需创建开销大的对象（懒加载） |
| **保护代理** | 权限控制 | 控制对原始对象的访问权限 |
| **智能引用** | 附加操作 | 在访问对象时添加额外行为（计数、锁等） |
| **本例** | 智能引用/装饰 | 通话前添加模仿功能 |

## 五、优缺点

### 优点
1. **开闭原则**: 无需修改原始类即可添加功能
2. **单一职责**: 代理类专注于控制逻辑，原始类专注于业务
3. **控制访问**: 可以在不改变原始对象的情况下控制访问
4. **延迟初始化**: 虚拟代理可以延迟创建开销大的对象

### 缺点
1. 增加了类的数量，系统可能变复杂
2. 代理层可能带来额外的性能开销
3. 需要为每个需要代理的类创建单独的代理

## 六、适用场景

- 需要控制对某个对象的访问（权限、频率等）
- 需要延迟加载重量级对象（虚拟代理）
- 需要为远程服务提供本地代表（远程代理）
- 需要在访问对象时执行额外操作（日志、缓存、计数）

## 七、代理模式 vs 装饰模式

| 特性 | 代理模式 | 装饰模式 |
|------|----------|----------|
| 目的 | 控制访问 | 增强功能 |
| 关注点 | 访问控制、延迟加载 | 动态添加功能 |
| 关系 | 代理"代表"真实对象 | 装饰器"包装"对象 |
| 创建 | 代理通常自己创建真实对象 | 装饰器从外部接收对象 |
| 本例特点 | DenDenMushi内部创建Speaker | DevilFruit从外部接收Teach |
