# 抽象工厂模式 (Abstract Factory Pattern)

## 一、模式概述

**意图**：提供一个接口来创建一系列相关或相互依赖的对象，而无需指定它们具体的类。

**别名**：Kit

**动机**：在某些系统中，需要创建一系列相关的产品对象（产品族）。例如GUI框架中需要创建不同风格的按钮、窗口、滚动条。如果为每种风格创建独立的工厂，会导致工厂数量爆炸。抽象工厂模式通过一个工厂接口来创建完整的产品族。

## 二、模式结构

```
+------------------+        +-----------------------------------+
| AbstractFactory  |        |         产品族层次结构              |
|------------------|        |                                   |
| + createShip()   |        |  ShipBody     Weapon      Engine  |
+--------+---------+        |    ^            ^          ^      |
         |                  |    |            |          |      |
+--------+---------+        | WoodBody       Gun       Human    |
|        |         |        | IronBody       Cannon    Diesel   |
Basic   Standard  Ultimate  | MetalBody      Laser     Nuclear  |
Factory  Factory  Factory   |                                   |
+------------------+        +-----------------------------------+
         |
         | createShip()
         v
    +---------+      产品族组合
    |  Ship   | = ShipBody + Weapon + Engine
    +---------+
```

**核心参与者**：

| 角色 | 类名 | 说明 |
|------|------|------|
| 抽象产品族 | `ShipBody`, `Weapon`, `Engine` | 各产品族的抽象接口 |
| 具体产品 | `WoodBody/IronBody/MetalBody`, `Gun/Cannon/Laser`, `Human/Diesel/Nuclear` | 具体产品实现 |
| 抽象工厂 | `AbstractFactory` | 声明创建各抽象产品的方法 |
| 具体工厂 | `BasicFactory`, `StandardFactory`, `UltimateFactory` | 创建具体产品族 |
| 客户端 | `Ship` | 聚合使用各产品族的产品 |

## 三、示例场景

以《海贼王》中的战船生产系统为背景：
- 不同等级的战船需要不同配置的船体、武器和动力系统
- **基础型工厂**：木船体 + 枪 + 人力驱动 = 基础战船
- **标准型工厂**：钢铁船体 + 自动机关炮 + 内燃机 = 标准战船
- **旗舰型工厂**：合金船体 + 激光武器 + 核能 = 终极战船
- 每个工厂创建的产品属于同一个产品族，保证产品之间兼容

产品等级结构 vs 产品族：

```
          ShipBody    Weapon    Engine      <- 产品等级 (抽象类型)
            |           |         |
  基础型:  WoodBody     Gun      Human       <- 产品族 (基础型)
  标准型:  IronBody    Cannon   Diesel       <- 产品族 (标准型)
  旗舰型:  MetalBody   Laser    Nuclear      <- 产品族 (旗舰型)
```

## 四、优缺点

**优点**：
1. 保证同一工厂创建的产品相互兼容（产品族内聚）
2. 客户端与具体产品类解耦，只通过抽象接口操作
3. 新增产品族时只需要新增具体工厂类，符合开闭原则
4. 隔离了具体类的生成，替换工厂即可切换整套产品系列

**缺点**：
1. 新增产品等级（如新增"雷达系统"）需要修改抽象工厂接口及所有具体工厂
2. 产品族的扩展遵循开闭原则，但产品等级结构的扩展违反开闭原则
3. 类的层次结构更加复杂

## 五、适用场景

- 需要创建一系列相关的产品对象（产品族）
- 系统需要独立于产品的创建、组合和表示
- 需要保证同一产品族的对象在一起使用时互相兼容
- 需要提供一个产品类库，暴露接口而非实现

## 六、与工厂方法模式的对比

| 维度 | 工厂方法 | 抽象工厂 |
|------|---------|---------|
| 创建范围 | 单个产品 | 产品族 (多个产品) |
| 工厂数量 | 每产品1工厂 | 每产品族1工厂 |
| 扩展方式 | 新增产品=新增工厂 | 新增产品族=新增工厂 |
| 复杂度 | 中 | 高 |
| 关系 | 抽象工厂常由工厂方法实现 | 使用工厂方法创建各产品 |

## 七、相关模式

- **工厂方法模式**：抽象工厂的每个方法通常都是工厂方法
- **单例模式**：具体工厂类通常实现为单例
- **原型模式**：抽象工厂可以用原型模式来实现产品创建
