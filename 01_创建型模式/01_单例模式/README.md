# 单例模式 (Singleton Pattern)

## 一、模式概述

**意图**：确保一个类只有一个实例，并提供一个全局访问点来获取这个实例。

**动机**：在软件系统中，有些对象我们只需要一个，例如配置管理器、数据库连接池、日志记录器、任务队列等。如果创建多个实例，会导致资源浪费、状态不一致等问题。单例模式通过私有化构造函数并提供静态访问方法来保证类的唯一性。

## 二、模式结构

```
+-----------------------+
|     Singleton         |
+-----------------------+
| - static instance     |
| - Singleton() private |
+-----------------------+
| + static getInstance()|
| + business methods    |
+-----------------------+
```

**核心参与者**：
- **单例类 (Singleton)**：负责创建自己的唯一实例，提供全局访问方法。

## 三、两种实现方式

### 3.1 饿汉模式 (Eager Initialization)

`TaskQueue` 类使用饿汉模式实现：

- **实现方式**：在类加载时（静态成员初始化阶段）立即创建实例
- **优点**：实现简单，无需考虑多线程同步问题
- **缺点**：即使程序从未使用该实例，也会创建，可能造成资源浪费
- **线程安全**：是的，因为实例在 main() 执行前就已创建

```cpp
class TaskQueue {
public:
    static TaskQueue* getInstance() { return &m_obj; }
private:
    TaskQueue() = default;
    static TaskQueue m_obj;  // 静态成员，类外初始化
};
TaskQueue TaskQueue::m_obj;  // 全局初始化
```

### 3.2 懒汉模式 (Lazy Initialization / Meyers' Singleton)

`S` 类使用懒汉模式实现（被原注释标注为"下面这种更好"）：

- **实现方式**：首次调用 `getInstance()` 时才创建实例
- **优点**：按需创建，避免资源浪费
- **线程安全**：C++11 起，静态局部变量的初始化是线程安全的
- **这是推荐的单例实现方式**

```cpp
class S {
public:
    static S& getInstance() {
        static S instance;  // C++11保证线程安全
        return instance;
    }
private:
    S() = default;
    S(const S&) = delete;
    S& operator=(const S&) = delete;
};
```

## 四、示例场景

以《海贼王》中巴基的账本升级为应用程序为背景：
- 需要一个全局唯一的任务队列来管理多线程任务
- 使用 `TaskQueue` 实现线程安全的添加和取出任务操作
- 两个线程并发访问同一个队列，验证单例在多线程环境下的正确性

## 五、优缺点

**优点**：
1. 严格控制唯一实例，保证全局状态一致性
2. 提供全局访问点，方便使用
3. 懒汉模式按需创建，节省资源

**缺点**：
1. 单例模式本质上是全局状态，增加了代码耦合度
2. 在多线程环境下，懒汉模式需要额外同步（C++11 已解决）
3. 单例类的测试比较困难（难以 mock）
4. 违反单一职责原则（既管理自身创建又管理业务逻辑）

## 六、适用场景

- 需要全局唯一对象的场景（配置管理、日志记录）
- 需要控制共享资源访问的场景（数据库连接池、线程池）
- 需要频繁创建和销毁的对象，使用单例提高性能

## 七、相关模式

- **抽象工厂、建造者、原型**：都可以用单例来实现工厂类
- **外观模式**：外观类通常设计为单例
