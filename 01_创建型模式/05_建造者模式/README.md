# 建造者模式 (Builder Pattern)

## 一、模式概述

**意图**：将复杂对象的构建与表示分离，使得同样的构建过程可以创建不同的表示。

**别名**：生成器模式

**动机**：当一个对象的创建过程非常复杂（需要多个步骤、多个部件组合），或者同一个构建过程需要创建不同表示的对象时，使用建造者模式可以将构建逻辑从产品类中分离出来。

## 二、模式结构

```
+---------------------+          +----------------------+
|      Director       |          |    ShipBuilder       |  (抽象建造者)
|---------------------|          |----------------------|
| - m_builder*        |  uses    | + reset()            |
| + setBuilder()      | -------> | + buildBody()        |
| + buildSimpleShip() |          | + buildWeapon()      |
| + buildStandardShip()|          | + buildEngine()      |
| + buildRegalShip()  |          | + buildInterior()    |
+---------------------+          +----------+-----------+
                                            |
                               +------------+-----------+
                               |            |            |
                       +-------+---+  +-----+-----+
                       | SunnyBuilder| |MerryBuilder|  (具体建造者)
                       |-------------| |-------------|
                       | - m_sunny*  | | - m_merry*  |
                       | + getSunny()| | + getMerry()|
                       +------+------+ +------+------+
                              |               |
                      +-------+--+     +------+------+
                      | SunnyShip |     | MerryShip  |    (产品)
                      +-----------+     +------------+
```

**核心参与者**：

| 角色 | 类名 | 说明 |
|------|------|------|
| 抽象建造者 | `ShipBuilder` | 定义构建产品各部分的接口 |
| 具体建造者 | `SunnyBuilder`, `MerryBuilder` | 实现构建步骤，构造具体产品 |
| 产品 | `SunnyShip`, `MerryShip` | 被构建的复杂对象 |
| 导演类 | `Director` | 控制构建流程，调用建造者方法 |

## 三、示例场景

以《海贼王》中的海贼船建造为背景：
- **桑尼号** (SunnyShip): 用 vector 存储零件列表
  - 神树亚当的树干 (船体)
  - 狮吼炮 (武器)
  - 可乐驱动 (动力)
  - 豪华内室精装 (内室)
- **梅利号** (MerryShip): 用 map 存储零件键值对
  - 船体: 优质木材
  - 武器: 四门大炮
  - 动力: 蒸汽机
  - 内室: 精装

导演类提供三种建造方案：
- **简约型**: 船体 + 动力
- **标准型**: 船体 + 动力 + 武器
- **豪华型**: 船体 + 动力 + 武器 + 内室

## 四、优缺点

**优点**：
1. 将复杂对象的构建与表示分离，客户端不必知道内部组成细节
2. 可以通过不同的建造者构建不同的产品
3. 导演类统一控制构建流程，确保产品按正确步骤构建
4. 符合单一职责原则，每个建造者只负责一种产品的构建

**缺点**：
1. 需要为每种产品类型创建对应的建造者，增加了类的数量
2. 如果产品之间的差异很大，不适合使用建造者模式
3. 建造者创建的产品需要有共同的构建步骤，否则抽象接口难以设计

## 五、适用场景

- 对象的构建过程非常复杂（多个步骤或部件）
- 需要分步骤创建对象，且步骤的顺序很重要
- 同样的构建过程需要创建不同的产品表示
- 需要隔离复杂对象的创建和使用

## 六、与其他模式的对比

| 模式 | 创建方式 | 关注点 |
|------|---------|--------|
| 建造者 | 分步骤构建 | 复杂对象的构建过程 |
| 抽象工厂 | 一步创建 | 产品族的创建 |
| 工厂方法 | 一步创建 | 单个产品的创建 |

## 七、相关模式

- **抽象工厂模式**：与建造者相似，但抽象工厂关注产品族的创建，建造者关注构建过程
- **组合模式**：建造者建造的产品常是组合对象
- **模板方法模式**：导演类通常使用模板方法来控制构建步骤
